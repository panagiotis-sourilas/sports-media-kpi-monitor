# Ingestion

Three files, three shapes.

| Source | Rows per month | What it looks like |
|---|---|---|
| `financial.csv` | ~600 | brand × month × P&L line, one number in local currency, one column for currency code |
| `budget.csv` | ~600 | same shape as financial, frozen once a year in January |
| `traffic.csv` | ~5,400 | brand × day × device, users / sessions / pageviews |

Everything lands in `gs://<bucket>/raw/<source>/<yyyy-mm-dd>_<source>.csv`. BigQuery Data Transfer picks up new files and appends to `raw.<source>`. That's it for ingestion — the modeling happens in dbt, not here.

## Why this shape

It's ELT, not ETL. We land the file raw and let dbt do the work. Ten years ago this was expensive because storage was expensive; now BigQuery storage is basically free and compute is on-demand. So we keep everything raw and reshape it on the way out. If we discover a business rule was wrong, we edit one dbt model and rebuild — we don't touch ingestion.

## Why not Fivetran

At CHF 300/month total infra, Fivetran's ~$1,500/month license costs more than everything else combined, and half our sources aren't in their catalog anyway. Long version in [`../docs/decisions/0001-why-not-fivetran.md`](../docs/decisions/0001-why-not-fivetran.md).

## The synthetic data

The production version reads from SAP, an Excel budget file, and GA4. This public version reads from CSVs generated by `synthetic_data/generate.py`. The generator:

- Runs with a fixed seed. Same output every time.
- Creates 24 months of financial + budget + daily traffic.
- Puts realistic seasonality into traffic (weekend spikes, tournament peaks).
- Deliberately drops some rows and mangles some currencies to prove the downstream models handle bad input.

Run it:

```bash
python synthetic_data/generate.py --months 24 --seed 42 --out raw/
```
