# Ingestion

## Sources

Three synthetic source shapes, mirroring what the production version pulls from:

| Source | Shape | Frequency | Real-world equivalent |
|---|---|---|---|
| **`financial.csv`** | Actuals per brand × month × P&L line | Monthly (2nd of month) | SAP financial export |
| **`budget.csv`** | Budget per brand × month × P&L line | Annual (frozen Jan) | Excel budget spreadsheet |
| **`traffic.csv`** | Daily users + sessions + pageviews per brand | Daily | GA4 (via native BQ export or Data API) |

## Landing pattern

Every source lands as a **CSV in a GCS bucket** under `raw/<source>/<YYYY-MM-DD>_<source>.csv`. BigQuery Data Transfer picks up new files and appends to `raw.<source>` tables.

This is the **modern "ELT" pattern**: land raw data with minimal transformation, do the real shaping in dbt.

## Why this pattern (not Fivetran)

See [`../docs/decisions/0001-why-not-fivetran.md`](../docs/decisions/0001-why-not-fivetran.md).

TL;DR: at our scale, managed EL tools would 5-10× the total infra cost to replace ~10-15 hours/month of engineering time. The math doesn't work.

## Synthetic data generator

`synthetic_data/` contains a reproducible fake-data generator. Everything in the public repo is generated by it — no real business data ever touches this repo.

```bash
python synthetic_data/generate.py --months 24 --seed 42 --out raw/
```

Generates:
- 24 months of financial + budget data (~2000 rows)
- 24 months × 6 brands × 30 days of traffic (~4300 rows)
- Realistic seasonality, brand-size heterogeneity, occasional data-quality issues (missing rows, currency edge cases) — because production data is never clean, and the report needs to handle that.
